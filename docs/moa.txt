
Mixture-of-Agents



项目主要做评审、推荐、建议、决策、头脑风暴高价值使用，也会做一些审核、审查、分析、测试中等价值的多agent工作。

启动方式：1.手动执行 2.关键词触发：moa模式、多人评审、多人委员会、对上面的分析做出建议、对上面的总结做出建议 3.在agent主模型遇到困难需要的时候它自己可以调用

使用场景：手动执行分析文档，对多个选项做出准确的决策，对审核结果或分析结果进行二次确认，上下文中的最后总结的评审

五模型委员会：当前agent为1人仲裁+另外4个agent

互动模式：当前主模型根据不同场景不同阶段不同工作，深入思考智能选择，几个agent有：角色扮演、开会讨论、独立回答、匿名互评、交叉审查、主席综合、仲裁模式

moa skill 有三种调用模式 看是分开命令做还是用参数做，你按用户体验好来设计

默认模式（手动调用） 顶配四个模型加一个当前agent设定的模型作为总领人

智能模式（默认调用） agent当前模型作为总领人，按场景、擅长、难易度、阶段，智能选择分配模型、人数、互动模式

自定义模式 指定人数（除当前agent设定模型作为仲裁人之外的人数，最多四人） 和 指定模型（人数和模型数人对应）

擅长：

Agent A 擅长逻辑推理  GPT-5.6 Sol 或 Claude Fable 5
Agent B 擅长代码     Claude Fable 5 或  Claude Opus 4.8
Agent C 擅长事实核查  Gemini 3.1 Pro Preview 或 Grok 4.5
Agent D 从用户体验或反方角度分析 Claude Fable 5 或 反方 Grok 4.5

阶段：

架构阶段，主要指的是分析需求和设计方案或 对需求文档、计划文档、开发文档、头脑风暴作出评审或建议

编码阶段，主要是指实现功能阶段出现的bug分析、问题分析、功能分析的加入和建议，主要是解决问题，不是全程开发。

测试阶段，主要是指运行真实使用流程，或对测试出来的问题报告时多人审核并给出修复建议

审查阶段，主要是对整个项目进行查找缺陷、安全问题和技术债，可以多agent参与

仲裁阶段，主要是指判断是否达到交付标准，或对有几种方案，有几个方向，有几个选择，有几个答案的情况下，给出准确建议


moa启动时调用多agent的分配：
当前使用的agent有的模型就用当前的如claude主模型用的Claude Fable 5，四人组中可以派个Claude Opus 4.8的agent,本机上又装了codex的，可以派个codex的GPT-5.6 Sol的agent
其它的agent可以调用openai或OPENROUTER的key来派Gemini 3.1 Pro Preview的 agent和Grok 4.5的agent

也可以用同一个强模型配上足够差异化的提示词。（后来的 Self-MoA 研究也印证了这一点:用同一个强模型配上足够差异化的提示词,效果往往好于混合多个弱模型——所以你不需要接入一堆不同模型,把角色提示词写出真实差异更重要。）

llm api支技openai和OPENROUTER格式接入，key放环境变量，环境变量有代理时调用llm优先走代理。默认模型都用最新顶级模型。

在保证质量的情况下要精炼语句，上下文适中，范围限制，token消耗性价比要高。
