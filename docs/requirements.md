## 1. 项目概述

**moa-skill** 是一个 Claude Code Skill：把当前主 agent 与最多 4 个异构模型委员组成"五模型委员会"（Mixture-of-Agents），在推理阶段通过"并行独立生成 → 结构化互动 → 证据驱动收敛"，对高价值判断类任务产出比任何单一模型更可靠的结论。

一句话定位：**给主 agent 一个随叫随到的专家委员会，而不是一条自动开发流水线。**

### 1.1 理论依据（摘要）

- **MoA（Wang et al., ICLR 2025）**：模型即使自己没有得出最佳答案，也能识别、整合和改进其他模型的答案（"协作性"）。效果来自 **质量 × 多样性**。
- **Self-MoA（arXiv:2502.00674）**：同一强模型 + 足够差异化的角色提示词，常优于混合多个弱模型 → 异构模型不可用时有可靠的降级路径。
- **Hermes MoA 2.0 成本结构**：委员只读剥离上下文的简报（无系统提示词、无工具 Schema），聚合者带完整上下文做最终决策，参考调用的 token 增量远小于主调用 → 委员会成本可控的关键。
- **已知局限**（需求层面必须对冲）：成本倍增、延迟、上下文膨胀、错误传播、弱模型污染、"假讨论"、停止条件难定。对应对策见 §8 非功能需求与 [design.md §9](design.md)。

### 1.2 与串行开发流水线的边界

MoA 的价值在"并行出多样性、再聚合"的**判断力密集节点**（评审、决策、仲裁、头脑风暴）。编码、修复这类强状态、强工具依赖的串行工作**不属于本 skill 范围**——本 skill 在编码/测试阶段只做"问题分析与建议"，不做并行写代码，不做全程自动开发。

### 1.3 差异化定位

对照 `moa推荐参考项目.md` 所列 7 个开源项目的实仓调研（[reference-adoption.md](reference-adoption.md)），本项目的差异点：

1. **三通道混合调度**：子代理（订阅内）+ 本地 CLI + API 混合派发、fallback 降级链、Self-MoA 主动/兜底双路径——7 个参考项目均为纯 API 或纯 CLI 单通道拓扑，无一混合，也无成本分层（订阅通道优先、API 补长尾）。
2. **仲裁人 = 当前 agent 带完整上下文**，且配统计块钳制 + 自查门 + 证伪检查对冲立场污染——与 MoA 论文"聚合者/提案者收益不对称定律"一致；参考项目的聚合者要么是再调一次 API（无上下文），要么无任何反幻觉钳制。
3. **成本自适应**：场景×难度×阶段三维路由 + L0/可机械验证双拒启 + dry-run 预演——参考项目全是单档全量运行（moa-x 每次 6–12 分钟、agent-review-panel 每次 $10–20）。
4. **可组合互动模式**：7 种模式拼装成"生成→精炼→收敛"流水线，按场景选装——参考项目各自固定 1–3 条流程。
5. **触发方式最全**：手动 + 关键词 + **主 agent 遇困自调（T3）**——参考项目全部仅手动触发。
6. **反群体思维纪律栈最完整**：盲审隔离、匿名互评、同源共识去重、谄媚计数器、三态弃权、孤例保护、LOW 置信退还决定权——每个参考项目各有其中 1–2 条，本项目是首个集成栈。
7. **中文优先双语**：报告正文跟随用户语言，参考项目全部英文单语。

## 2. 价值场景

### 2.1 高价值（核心场景，必须做好）

| 场景 | 说明 | 典型输入 |
|---|---|---|
| 评审 | 对方案、架构、PRD、代码、文档做多视角对抗评审 | "对这份设计文档做多人评审" |
| 决策 | 多个选项/方向/答案中给出准确建议 | "PostgreSQL 还是 MongoDB？拆微服务还是加缓存？" |
| 推荐/建议 | 对分析结果、总结给出改进建议 | "对上面的分析做出建议" |
| 头脑风暴 | 发散差异化点子并策展 | "给这个产品想差异化功能" |

### 2.2 中价值（支持场景，够用即可）

审核、审查、分析、测试问题报告的多人复核与修复建议、审核/分析结果的二次确认、上下文最终总结的评审。

### 2.3 非目标（明确不做）

- 简单问答、低延迟对话（单模型更划算，编排器应拒绝启动并说明）
- 多委员并行写同一份代码
- 自动提交/发布/CI 操作
- 全程自动开发闭环（那是另一个 skill 的职责）

## 3. 触发方式（三种）

| # | 方式 | 说明 |
|---|---|---|
| T1 | 手动执行 | `/moa` 命令（含参数），或明确指示"用 moa 分析这份文档" |
| T2 | 关键词触发 | 用户消息含触发词即建议/进入 moa：`moa模式`、`多人评审`、`多人委员会`、`委员会`、`对上面的分析做出建议`、`对上面的总结做出建议`、`多模型`、`第二意见`、`交叉验证` 及英文 `moa / council / multi-model review` |
| T3 | 主 agent 自主调用 | 主模型遇到困难时自行召集：多方案难以取舍、对自身结论置信度低、发现自己反复失败需要外部视角。SKILL.md description 中写明此授权 |

T2/T3 触发时，若任务难度为 L0（见 §7），skill 应告知"此问题无需委员会"并由主模型直接回答——防止关键词误触发烧钱。

## 4. 委员会构成

**固定结构：当前主 agent 为 1 名仲裁人/主席 + 最多 4 个委员席位。** L3 高风险任务允许 D 席拆分为 D1(UX) + D2(反方) 两次调用（总委员调用 ≤5，席位仍计 4）；D 席拆分适用于 auto/full 模式，custom 模式经 `--models` 显式列出等价配置。

仲裁人（当前 agent）持有完整对话上下文与工具，负责：构建简报、召集委员、综合/仲裁、向用户报告。委员在**生成阶段**是无状态盲审者，只看到自包含简报；**精炼阶段**按互动模式受控看到匿名化的他人意见（其中"开会讨论"是盲审原则的显式例外，见 §6）。

### 4.1 委员专长矩阵（默认顶配）

| 委员 | 专长 | 首选模型 | 备选 | 推荐运行设置 |
|---|---|---|---|---|
| A | 逻辑推理、方案权衡、根因分析 | GPT-5.6 Sol | Claude Fable 5 | `reasoning_effort: xhigh`（重大决策 `max`） |
| B | 代码理解、工程实现、重构 | Claude Opus 4.8 | Claude Fable 5 / GPT-5.6 Sol | `effort: xhigh` |
| C | 事实核查、证据验证 | Gemini 3.1 Pro Preview | Grok 4.5 / GPT-5.6 Sol + Web Search | 尽可能开启搜索接地；无搜索工具时降级为"一致性核查"并显式声明 |
| D | 用户体验 / 反方红队 | Grok 4.5（反方） | Claude Fable 5（纯 UX） | `reasoning_effort: high`；L3 高风险可拆 D1+D2（见 §4） |

选择原则：**覆盖 OpenAI / Anthropic / Google / xAI 四个独立模型家族**，最大化错误去相关。模型 ID 迭代快，正式运行前须对照提供商/openrouter.ai/models 核对（配置层解耦，见 [design.md §7](design.md)）。

> 偏离说明：B/D 席首选顺序相对 moa.txt（两席均把 Claude Fable 5 列首位）做了有意调整——仲裁人已由 Claude 家族主模型担任，委员席优先保证模型家族多样性；moa.txt 第 52 行亦明示"四人组中可以派个 Claude Opus 4.8 的 agent"。用户可经 custom 模式随时覆盖。

### 4.2 事实核查委员的硬约束

> 每项重要事实必须给出来源，优先级：官方文档 > 原始论文/代码 > 权威机构 > 二手报道；找不到可靠来源时标注"未验证"，禁止用常识补全。

无搜索能力的"事实核查委员"只是另一个会幻觉的模型——该约束写入角色提示词（见 design.md §6 角色契约）。

## 5. 三种调用模式

**产品决策：单一入口 `/moa` + 子命令/参数**，而非三条独立命令。理由：三条命令会使触发面翻三倍、关键词路由混乱；单入口下参数对老手渐进暴露。默认档位遵循 moa.txt 原文：**手动执行默认顶配（full），关键词/自调触发默认智能（auto）**。

| 模式 | 调用 | 行为 |
|---|---|---|
| **顶配模式 `full`**（手动默认，即 moa.txt 的"默认模式"） | `/moa <材料>`、`/moa full <材料>` | 固定 4 名顶配异构委员（§4.1 首选列）+ 当前 agent 仲裁，互动流水线仍按场景选择 |
| **智能模式 `auto`**（T2/T3 默认，即 moa.txt 的"智能模式"） | `/moa auto <材料>`、关键词触发、agent 自调 | 编排器按 **场景 × 难度 × 阶段** 智能决定：委员人数（1–4）、模型分配、互动模式流水线、精炼轮数 |
| **自定义模式 `custom`** | `/moa custom --members 2 --models "gpt-5.6-sol,grok-4.5" <材料>` | 用户指定委员人数（1–4，不含仲裁人）与模型（数量一一对应）；可选 `--interaction` 指定互动模式。**允许 `--models` 重复同一强模型**（自动为各席配差异化角色 = 主动 Self-MoA，见 §9.2）；模型 → 通道解析规则见 design.md §4.4 |

通用可选参数：`--refine-rounds 0|1|2`（精炼轮数，默认按难度：L1=0、L2≤1、L3≤2）、`--dry-run`（预演：展示委员构成与成本估算，不实际调用）、`--interaction <mode>`（强制互动模式）。

## 6. 互动模式（7 种）

互动模式是**可组合的流水线阶段**，分三类：

**生成阶段**（选一）
1. **独立回答**：各委员并行、互相隔离，直接回答简报问题。
2. **角色扮演**：各委员戴上对抗角色/人格透镜（安全审计员、可行性质疑者、用户代言人、成本魔鬼代言人；头脑风暴用发散人格）后独立生成。多样性主要来源。

**精炼阶段**（0–2 个）
3. **匿名互评**：每位委员看到**匿名化**的他人意见，逐条同意/反驳/合并，输出修订意见。匿名防止"大牌模型"意见压人——只认论据不认出处。
4. **交叉审查**：指派每位委员专门攻击另一位的方案，找致命弱点（对抗式论证，决策场景质量最高）。
5. **开会讨论**：顺序发言、后发者可见先前发言，多轮。成本最高、有从众风险，仅 L3 复杂争议且用户明确要求时使用。

**收敛阶段**（选一，由当前 agent 执行）
6. **主席综合**：仲裁人按硬规则合成报告——共识置顶、分歧原样保留、禁止和稀泥、禁止淡化 blocker、禁止编造问题（硬规则全文见 design.md §8）。
7. **仲裁模式**：在主席综合基础上给出裁决，裁决形式依场景——交付/放行判断：`PASS / PASS_WITH_RISK / BLOCKED / INCONCLUSIVE`；多选项决策：`RECOMMEND <选项>` + 置信度，或 `INCONCLUSIVE`。两种形式均须附**结论失效条件**。

收敛封顶条款（两条，优先级高于给出结论的冲动）：
- **全体否决熔断**：全部委员否决所有候选时，收敛必须 STOP，输出 `REJECTED` 并把问题退回用户，禁止硬凑方案（采纳自 moa-x 双 reject_all 熔断）。
- **低置信退还**：委员根本性分歧且整体置信度低时，固定输出"此决定应由你本人做出，不应委托给委员会"，逐一列出各委员立场与自报置信度（采纳自 agent-council）。

### 6.1 场景 → 默认流水线映射

| 场景 | 生成 | 精炼 | 收敛 |
|---|---|---|---|
| 评审/审查/审核 | 角色扮演（评审角色） | L2+：匿名互评 | 主席综合 |
| 决策/推荐 | 角色扮演（各认领一个选项，须论证己方最强+指出对方致命弱点；选项数与席位数不等时的认领规则见 design.md §6） | 交叉审查 | 仲裁模式（对比矩阵+置信度+失效条件） |
| 头脑风暴 | 角色扮演（发散人格，高 temperature） | 无 | 主席综合（策展：孤例保护、禁止磨平棱角） |
| 二次确认 | 独立回答 | 无 | 主席综合（共识/分歧） |
| 总结评审 | 独立回答或角色扮演 | L3：匿名互评 | 主席综合 |

## 7. 难度分级与资源分配

难度由编排器评估（材料复杂度、错误代价、分歧空间），用户可覆盖。

| 难度 | 定义 | 委员数 | 精炼轮 |
|---|---|---|---|
| L0 | 简单问答、答案基本唯一；**或可机械验证的客观问题**（算术、事实检索——实测 MoA 收益为负，应直接验证而非开会） | **0（不启动，主模型直答或直接验证）** | — |
| L1 | 局部、低风险、可逆的判断 | 1–2 | 0 |
| L2 | 常规方案评审、普通技术选型 | 3 | 0–1 |
| L3 | 架构决策、不可逆选择、交付仲裁、安全审计 | 4（可 D 席拆分，总调用 ≤5，见 §4） | 1–2 |

## 8. 阶段定义（auto 模式的路由维度之一）

| 阶段 | 内容 | 默认委员侧重 |
|---|---|---|
| 架构阶段 | 需求分析、方案设计；对需求/计划/开发文档、头脑风暴做评审或建议 | A + D + C |
| 编码阶段 | 实现期的 bug 分析、问题分析、功能建议（解决问题，不是全程开发） | B + A |
| 测试阶段 | 真实使用流程验证；测试问题报告的多人审核与修复建议 | B + D |
| 审查阶段 | 全项目查缺陷、安全问题、技术债 | B + C + 安全角色 |
| 仲裁阶段 | 是否达到交付标准；多方案/多方向/多答案的准确建议 | A + C + D（全员亦可） |

## 9. 模型接入需求

### 9.1 三通道调度（按优先级探测）

| 通道 | 适用 | 说明 |
|---|---|---|
| CH1 本地子代理 | Claude 系模型 | 经 Claude Code Task/Agent 机制派发（如 Opus 4.8 子代理），走现有订阅，无需额外 key |
| CH2 本地 CLI | 已安装的 codex CLI（GPT-5.6 Sol） | 探测到可执行文件即可用，走该 CLI 自身鉴权。其他 CLI（如 gemini）为扩展点，v1.0 不承诺 |
| CH3 HTTP API | 其余模型 | **openai 兼容协议 + openrouter 协议**双支持；OpenRouter 一个 key 可调所有厂商 |

### 9.2 硬性要求

- API key 一律走环境变量（`OPENROUTER_API_KEY`、`OPENAI_API_KEY`、`GEMINI_API_KEY`、`XAI_API_KEY`），禁止写入文件/日志/报告。
- 检测到 `http_proxy` / `https_proxy` / `no_proxy` 环境变量时，LLM 调用**优先走代理**，本机与 no_proxy 域名直连；运行日志提示代理状态。
- 默认模型全部使用当期各家顶级模型；模型 ID 只存在于配置层，便于随迭代更新。
- **仲裁人永不降级**：fallback 链只降委员席位，仲裁人始终是当前最强 agent——弱模型当提案者可以，当聚合者会灾难性拉胯（MoA 论文 Table 4 + togethercomputer/moa issue #41 实证）。
- **Self-MoA 两种路径**（同一强模型 × 足够差异化的角色提示词，效果常优于混合弱模型——moa.txt 明示的正面能力，不只是兜底）：
  - **主动**：custom 模式 `--models` 重复同一模型，或 `--self-moa` 开关；系统自动为各席分配差异化角色，报告中注明 Self-MoA 构成。
  - **兜底**：无任何外部通道可用时自动提议/执行——当前模型分回合独立扮演各委员（每回合只依据简报+角色契约，刻意不参考此前回合产出），完成后**必须向用户声明**：这是单模型多视角模拟，只有角色分化收益，没有跨模型错误去相关收益。

## 10. 输出要求

1. **报告正文**（给人读）：结论摘要（含裁决与置信度）→ 高置信问题（≥2 位委员独立提出，注明提出者）→ 单一来源问题 → **待人工裁决的分歧**（附双方论据，禁止折中）→ 各委员意见摘要 → 固定免责声明（"各模型训练数据存在重叠，全员未发现不代表不存在问题；分歧点需人工裁决"）。
2. **结构化数据**（JSON）：委员逐条意见、severity/confidence、统计（verdict 计票、按严重度问题数、平均置信度），供程序化处理与复核。
3. 决策场景额外输出：决策摘要表（问题/推荐/置信度/核心依据/下一步）、逐项的候选方案-评价标准-风险-**结论失效条件**-回退方案；多项决策须给出依赖关系与实施顺序。
4. 呈现报告时，仲裁人必须主动指出"待人工裁决的分歧"一节。

## 11. 非功能需求

| 维度 | 要求 |
|---|---|
| 成本 | 委员读剥离上下文的紧凑简报；委员输出限 `max_tokens_member`（默认 3000）；精炼轮数 ≤2；L0 拒绝启动；`--dry-run` 先给用户看成本估算。目标：L2 默认配置（3 委员、≤1 精炼轮）总 token ≤ 基线 7 倍；**基线** = 同一简报喂单个同级模型一次直答（同 max_tokens）的实测 token 数。**M4 实测（2026-07-11，见 [moa-reports/cost-m4/COST-NOTE.md](../moa-reports/cost-m4/COST-NOTE.md)）**：基线 = 2030 tok/席·轮；实测 2 席 + 1 精炼轮 = **4.79×**；精炼轮 prompt 携前轮全文，per-seat 膨胀 1.39×。**按用户实付 token 计**（默认 config 中 A=codex CLI、B=子代理走订阅不计费，仅 C/D 两席 CH3 计费）实付倍数 ≈ **4.79× ≤ 7× ✓**。**警示**：若四席全配 CH3 且开 1 精炼轮，总算力 ≈ 9.6×，超 7× 目标——降本手段：L2 默认 refine=0（4.0×）／保留 ≥2 订阅席／v1.1 精炼只传对方结论摘要而非全文。usage 捕捉已落地 `stats.token_usage` 与 `stats.r1.token_usage`（只计 CH3 计费席）。 |

## 12. 验收标准（v1.0）

1. `/moa`（=full）、`/moa auto`、`/moa custom` 三入口可用；`--dry-run` 输出委员构成、通道、代理状态与成本估算。
2. auto 模式对 §6.1 五类场景选出正确的默认流水线（每场景 ≥2 条固定正例）；L0 闸门用固定用例集（≥10 正例 / ≥10 负例）判定，通过线各 ≥9/10。
3. 三通道均有真实调用成功案例；任一通道缺失时自动降级并在报告中声明实际使用的委员阵容。
4. 聚合硬规则生效：构造含分歧的用例，报告须保留双方论据且 blocker 不被降级。
5. 委员故障注入（超时/非法 JSON/全挂）分别触发重试、JSON 修复、中止三种正确行为。
6. Self-MoA 降级路径可运行且报告含显式声明。
7. 三组真实验证：一次文档评审、一次多选项决策、一次头脑风暴，输出完整双格式报告。
8. 敏感信息提示、代理优先、key 不落盘经人工核查通过。
