这里有几个值得注意的设计要点。提议者和聚合者的分工基于模型各自的长处:有些模型擅长产出多样、信息量大的候选答案,有些则擅长综合与提炼,同一个模型也可以在不同层扮演不同角色。它和 Mixture of Experts(MoE)虽然名字相似,但完全不是一回事——MoE 是模型内部架构层面的稀疏专家子网络加门控路由,需要联合训练;而 MoA 是模型级别的协作,整个 LLM 作为智能体通过自然语言接口交互,无需改动任何权重。原始论文中,仅用开源模型组合的 MoA 在 AlpacaEval 2.0、MT-Bench 等基准上超过了当时的 GPT-4o,展示了"以小博大"的潜力。

**应用方向**主要集中在几个方面。一是质量提升与成本优化:用多个较小或开源模型的组合,达到甚至超越单个大型闭源模型的效果,在推理密集、指令遵循、开放问答等任务上尤其有效。二是合成数据生成与模型评估:MoA 式的多模型协作被用来产出更高质量的训练数据,或作为更稳健的"LLM 评审团"。三是智能体系统设计:MoA 的思想被吸收进各类 agent 框架,作为多模型协同、答案交叉验证的一种模式。四是衍生变体研究,比如 Self-MoA(用同一个强模型多次采样再聚合,有研究发现在某些场景下比混合多个较弱模型效果更好,揭示了"多样性 vs 单体质量"的权衡)、稀疏化 MoA 以降低调用开销等。

它的主要**局限**也很直接:多层串行调用带来显著的延迟和成本,首个 token 的响应时间较长,因此更适合对质量敏感、对时延不敏感的离线或批处理场景,而不太适合实时对话。如果你想深入某个方向,比如 MoA 的提示词具体怎么写,或者它与其他集成方法(如投票、best-of-N)的对比,我可以展开讲。
