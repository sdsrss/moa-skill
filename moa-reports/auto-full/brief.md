# 决策委员会简报:新一代 API 网关选型(自研 / Kong / Envoy)

## 背景
一个中型 SaaS,当前用一套 5 年前自己写的 Nginx+Lua 网关(约 6k 行 Lua),承担鉴权、限流、路由、灰度。痛点:Lua 生态老化、无人愿维护、加新特性慢、可观测性差。计划 6~9 个月内换掉,这是**多年一次、迁移代价高**的基础设施决策。

## 待决策问题
下一代网关选哪条路?
- **A. 继续自研**(换 Rust/Go 重写,完全自控)
- **B. Kong**(OpenResty 系,插件生态成熟,有企业版)
- **C. Envoy**(CNCF,xDS 动态配置,service mesh 友好)

## 已知约束
- 团队 8 人后端,2 人有较深网关经验;无专职 SRE/平台团队。
- 现有流量峰值 ~4 万 RPS;90 天内要平滑迁移,不能停服。
- 已在用 Kubernetes;中期有 service mesh 打算但未定。
- 预算有限,倾向开源自托管,不排斥企业版关键支持。
- 需保留:JWT 鉴权、按租户限流、金丝雀发布、审计日志。

## 委员会问题
1. 在上述约束下,哪条路整体最优?给对比与推荐。
2. 每个选项**被低估的隐藏成本**是什么(迁移、学习曲线、运维、锁定)?
3. 有没有 ≤1 天能显著降低不确定性的 spike?
