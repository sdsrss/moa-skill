# 评审简报:给内部 CLI 工具加"配置热重载"的设计方案

## 背景
我们有一个常驻的内部 CLI 守护进程(Python,单进程,~8k 行),启动时读一次 `config.yaml`。运维反馈:改配置要重启进程,重启会中断正在跑的任务(平均每次损失 3–5 分钟)。现提议加"配置热重载"。

## 待评方案(完整)
1. 主线程注册 `SIGHUP` 信号处理器;收到 SIGHUP 时重新读取 `config.yaml`,解析成新 dict,**整体替换**全局单例 `CONFIG`。
2. 各业务模块通过 `from app.config import CONFIG` 在函数内部**每次访问时**读 `CONFIG[...]`,不缓存到局部变量。
3. 解析失败(YAML 语法错/校验不过)时:打日志、丢弃新配置、保留旧 `CONFIG`,不中断进程。
4. 不加锁——"Python 有 GIL,dict 整体替换是原子的,读到的要么全旧要么全新,不会读到半个"。
5. 不做字段级 diff,不通知模块"哪些变了";模块自然在下次访问时读到新值。

## 已知约束
- 单进程,但内部用了线程池(最多 16 个 worker 线程并发跑任务)。
- 有些配置项(如 `db_pool_size`、`log_level`)被模块在启动时读取后缓存进了自己的对象,不走每次访问。
- 需要在不改动业务模块调用点的前提下尽量落地。
