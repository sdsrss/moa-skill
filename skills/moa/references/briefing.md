# 简报（黑板）写法

委员是无状态盲审者——**只能看到你写进简报的内容**，看不到对话历史、系统上下文、工具记录。简报质量直接决定评审质量，这是 MoA 架构最大弱点的弥补点。

把简报存为 `moa-reports/<run>/brief.md`，作为 `moa.py generate --input` 的输入。

## 必含五段

### 1. 背景（3–8 句）
项目是什么、面向谁、当前处于什么阶段。让一个从没见过这个项目的人能进入状态。

### 2. 待评对象本体
方案/代码/决策/文档的**完整内容**，不要只给摘要。这是委员唯一的评审依据——省略即等于让委员盲评。

### 3. 已知约束
预算、时限、技术栈、团队规模、不可更改的前提。委员据此判断建议是否落地。

### 4. 明确的委员会问题
你希望委员回答什么。越具体越好：
- 评审："这个架构能否支撑 10 倍流量？有哪些安全与可维护性风险？"
- 决策：列出**全部**候选选项及各自已知信息，问"在给定约束下推荐哪个，为什么"。

### 5. 范围与工作量夹层
- `out_of_scope`：明确不评什么，防止委员昂贵地游走到无关区域。
- 勘探预算（带工具通道适用）：`max_file_reads` / `max_searches` / `max_minutes` 上限防跑飞；引用/证据下限防敷衍。

## 自包含性检查（写完自检）

- [ ] 不含"上文/刚才/之前讨论的/如前所述"等对话内指称——委员看不到那些。
- [ ] 待评对象是完整原文，不是"见某文件"或摘要。
- [ ] 评审问题是疑问句，不是"随便看看"。
- [ ] 决策场景已列全部选项（含"维持现状"）。
- [ ] 缺任何关键信息 → 先问用户，不要自己脑补。

## 敏感信息提示

简报会被发送到 config 中所有第三方模型提供商。若含密钥样式、内部域名、个人信息、未公开的商业数据，正式运行前提醒用户，或改用纯本地通道（CH1/CH2）配置。key 本身绝不写进简报。

## 模板

```markdown
# MoA 简报：<一句话主题>

## 背景
<3-8 句>

## 待评对象
<完整本体>

## 已知约束
- <约束1>
- <约束2>

## 委员会问题
1. <具体问题>
2. <具体问题>

## 范围
- out_of_scope: <明确不评的部分>
- 勘探预算: max_file_reads=12, max_searches=6, max_minutes=4
```
