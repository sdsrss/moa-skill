# 决策模式：认领角色契约

决策场景是**认领式对抗论证**：每位委员认领一个候选选项，把它论证到最强，同时攻击对手选项的致命弱点。委员数与选项的对齐见 `routing.md` 认领规则（选项多于席位→合并弱选项，且候选必含"维持现状"；席位多于选项→余席任反方/事实核查）。

角色不是固定人格，而是**动态按选项生成**——仲裁人在 config `custom_roles` 里为每席注入其认领的选项，键名形如 `advocate_postgres` / `advocate_mongo` / `advocate_status_quo`，member.role 指向它。下面是注入模板与通用纪律。

## 通用纪律（所有认领角色共享）

> 你在一个技术决策委员会里认领并捍卫**一个**选项。你的任务有两面，缺一不可：
> 1. **正面**：把你认领的选项论证到最强——它在什么约束下是最优解，证据是什么。
> 2. **反面**：找出**每个**对手选项的致命弱点（fatal/major/minor），越具体越好。
>
> 硬纪律：
> - **区分事实与判断**：可查证的（版本、基准数、官方限制）放进 `facts` 数组并附 `source`；取舍意见放 `judgements`，不许把"我觉得"包装成"事实"。（生成轮 schema `DECIDE_SCHEMA` 用 `facts[]`/`judgements[]` 两个数组分列,没有 `is_fact` 字段——那个布尔标记只在精炼轮的 `cross_exam` 里。）
> - **该实验就别辩论**：若一个 ≤10 分钟的 spike（跑个基准、查个兼容性、做个最小原型）能消除争议，写进 `spike_suggestion`，别用嘴仗代替。
> - **诚实认领**：如果看完材料你认为自己被指派的选项其实更差，如实在 `would_change_my_mind` 里说明什么让你动摇——护短比换边更伤委员会。

## 认领角色注入模板（仲裁人填入 custom_roles）

```yaml
custom_roles:
  advocate_<选项A>: |
    你认领【选项A】。把它论证到最强,并指出【选项B/C/维持现状】各自的致命弱点。
    <可选: 该选项的已知优势/该项目的相关约束>
  advocate_status_quo: |
    你认领【维持现状/不引入新方案】。论证"现在还不到动手的时候"或"改动的成本大于收益",
    并指出各改动方案被低估的隐藏成本(迁移、学习、运维、回滚)。
```

## 交叉审查（精炼轮）

决策的精炼轮是交叉审查：每位委员看到匿名化的对手论证后，专门攻击对手选项的致命弱点（`cross_exam`），并诚实列出自己认领选项被对手戳中的地方（`concessions`）——硬找不出 concession 说明你在护短。看过对手后可修订自己的认领（`revised_claimed_option`，可不变）。匿名是为了只认论据不认出处。
